# 漏洞挖掘质量优化规划

本文面向后续持续优化，重点解决两个问题：候选漏洞质量不高，以及开发过程中积累的冗余代码、过期方案和职责冲突。

## 1. 当前判断

当前系统主架构是合理的：用规则和工具保证覆盖，用 LLM 做策略、语义判断、验证方案和报告解释。这比完全 ReAct 式 LLM 自主浏览仓库更可控，也更适合安全审计对可追溯证据的要求。

真正的问题不在“是否应该调用工具”，而在漏洞挖掘阶段的证据门槛不够硬。现在系统能够广泛发现危险 API、工具告警、依赖漏洞和配置风险，但会把弱信号过早包装成 finding，导致报告里候选数量多、真实可验证漏洞少。

## 2. 当前挖掘链路

```text
ToolRunner / rules / MiningDirector
  -> DangerousFunctionLocator
  -> SliceAnalyzer
  -> CandidateGenerator
  -> ClueAggregator
  -> VulnerabilityClassifier
  -> VerificationAgent
```

LLM 参与点：

| 阶段 | 当前作用 |
|---|---|
| `MiningDirector` | 浏览仓库，输出关注目录、优先函数、工具选择和验证建议 |
| `CandidateGenerator` | 对部分工具型 slice 生成候选 |
| `LLMCandidateReviewer` | 主要复核依赖和配置类噪声 |
| `VerificationAgent` | 生成验证 recipe、PoC/runbook、局部验证方案 |
| `ReportWriter` | 生成中文报告描述 |

当前 LLM 没有系统性承担“源码候选有效性闸门”的职责，这是质量问题的核心之一。

## 3. 主要问题

### 3.1 信号类型混杂

依赖漏洞、secret、供应链配置、Semgrep/Bandit 工具提示和源码危险函数都被包装进同一条 `DangerousFunction -> Slice -> Candidate -> Finding` 流程。

这会造成两个问题：

- 报告看起来 finding 很多，但很多不是源码漏洞；
- 后续验证逻辑需要不断判断哪些 finding 可以动态验证，复杂度增加。

### 3.2 危险函数命中门槛过低

规则命中 `memcpy`、`eval`、`subprocess`、`execute` 等危险 API 后，很多情况下可以直接进入 fallback candidate。危险 API 出现只能说明“值得检查”，不能说明“存在漏洞”。

后续应区分：

| 类型 | 含义 | 是否可直接进入 finding |
|---|---|---|
| `signal` | 工具或规则发现的线索 | 否 |
| `context_slice` | 危险点附近上下文 | 否 |
| `taint_slice` | source 变量进入 sink 参数 | 可进入候选 |
| `finding` | 经过静态/LLM/工具证据确认的问题 | 是 |

### 3.3 切片分析不够强

当前 `SliceAnalyzer` 主要是局部上下文摘取加正则推断，不是完整程序切片。它可以找函数范围、source、sink、guard、sanitizer 和简化 call chain，但不能可靠证明：

- 外部输入真的进入 sink；
- sink 参数依赖 source；
- 入口函数真实可达；
- 中间没有有效 sanitizer；
- 跨函数、结构体字段、对象状态传播成立。

这会导致“同一个上下文里同时出现 source 和 sink”被误认为存在数据流。

### 3.4 LLM 介入位置偏后

LLM 现在主要做策略和验证方案，不稳定参与源码候选过滤。规则型候选常走 deterministic fallback，LLM reviewer 也没有覆盖所有源码候选。

后续应让 LLM 在源码候选进入 finding 前做结构化 triage，但只能基于系统提供的证据判断，不能凭空补链路。

### 3.5 工具结果过度提升

Semgrep、Bandit、cppcheck、clang-tidy 等工具输出有不同语义。有些是强安全规则，有些只是 lint 或最佳实践提示。当前工具结果容易被统一转成 candidate，导致弱告警进入报告。

应将工具规则分级：

- 强安全 taint 规则；
- 安全相关 bug 规则；
- lint/quality 规则；
- 依赖/配置/secret 规则。

只有前两类可以进入源码漏洞候选。

### 3.6 噪声文件过滤不足

测试结果显示，minified JS、bundle、静态第三方库、编辑器/播放器前端资产会产生大量误报。

后续默认降权或排除：

```text
*.min.js
*.bundle.js
*.map
dist/
build/
coverage/
static/vendor/
public/vendor/
assets/vendor/
third_party/
examples/
samples/
tests/
fuzz/
```

测试、样例和 fuzz 目录不一定完全删除，但默认不应进入 top finding，除非用户选择深度模式或 LLM 明确标记为真实产品入口。

## 4. 优化目标

### 4.1 质量目标

1. 源码 finding 必须有明确 source、sink 和变量级关联。
2. 工具/规则命中默认只是 signal，不直接等价于漏洞。
3. 依赖、配置、secret 与源码漏洞分流展示、分流统计、分流验证。
4. LLM 必须参与源码候选 triage，但 LLM 失败不能默认确认。
5. 报告优先展示高证据 finding，弱信号进入“待人工复核”或调试区。

### 4.2 工程目标

1. 清理旧架构代码和重复 fallback。
2. 明确模型命名和数据契约。
3. 保留 mining debug，便于定位每个 finding 的来源。
4. 增加质量回归测试，避免重新引入噪声。

## 5. 分阶段实施计划

### 阶段 A：评估基线与报告治理

目标：先建立可重复评估体系。

任务：

1. 为 `audit-report.json` 和 `mining-debug.json` 增加 schema version。
2. 编写报告质量统计脚本，输出每个项目的 anchors、slices、candidates、findings、验证结论、source 缺失率、sink 缺失率、call chain 长度。
3. 在前端和脚本中标记 legacy report，避免旧架构报告干扰当前质量判断。
4. 将 dependency、config、secret 统计与 source_code finding 分开。

验收：

- 能一眼看出一个报告中真正源码 finding 的数量；
- 能按项目比较优化前后的噪声比例；
- 旧报告不会被误认为当前流水线结果。

### 阶段 B：信号分流

目标：把工具/规则信号和最终漏洞候选拆开。

任务：

1. 新增或重命名数据层概念：
   - `Signal`
   - `ContextSlice`
   - `TaintSlice`
   - `CodeCandidate`
   - `Finding`
2. `DangerousFunctionLocator` 输出更准确的 `Signal`，不再把所有内容称为危险函数。
3. dependency、config、secret 单独进入对应报告区，不再走源码切片和动态验证路径。
4. 工具结果增加规则强度字段，如 `security_taint`、`security_bug`、`lint`、`config`、`sca`、`secret`。

验收：

- SCA/config/secret 不再污染源码 finding 排名；
- source_code finding 数量下降，但证据质量提升；
- 依赖和配置风险仍能在报告中看到。

### 阶段 C：文件过滤与噪声降权

目标：减少明显无效输入。

任务：

1. 增加统一的 `PathPolicy`，由 Recon、ToolPlanner、DangerousFunctionLocator、SliceAnalyzer 共享。
2. 默认排除 minified、bundle、map、生成文件和第三方静态资产。
3. examples/tests/fuzz 进入低优先级池，不默认进入 top finding。
4. 在 mining debug 里记录被过滤原因和数量。

验收：

- maccms10 不再从 `*.min.js`、ueditor、播放器静态资产生成高优先级源码漏洞；
- 报告中显示过滤统计，便于解释为什么跳过某些目录。

### 阶段 D：变量级切片

目标：把当前“局部上下文切片”升级为“变量级 source-to-sink 证明”。

任务：

1. `ProgramSlice` 增加结构化字段：
   - `source_variable`
   - `sink_arguments`
   - `assignments`
   - `aliases`
   - `parameter_flow`
   - `field_flow`
   - `reachable_entry`
   - `guards_on_path`
   - `sanitizers_on_path`
   - `flow_confidence`
2. Python 使用 AST 追踪函数参数、赋值、调用参数、request 对象。
3. JS/TS 使用 tree-sitter 或 TypeScript AST 追踪 Express/Koa/Fastify 请求对象到 sink。
4. PHP 追踪 `$_GET`、`$_POST`、`$_REQUEST` 到数据库、命令、文件、模板 sink。
5. C/C++ 先完成函数内变量依赖图，再扩展到调用图和 parser entry。

进入高优先级候选的最低条件：

```text
source_variable exists
sink_argument depends on source_variable
no effective sanitizer observed
reachable_entry exists or parser/CLI/runtime entry exists
```

验收：

- 同函数内 source 和 sink 无变量关联时不能生成高优先级 finding；
- C/C++ `memcpy(dst, src, len)` 需要证明 `src` 或 `len` 可被外部输入影响；
- 报告中的触发链路能展示变量传播过程。

### 阶段 E：LLM 源码候选 triage

目标：让 LLM 成为源码 finding 前的语义审查环节。

任务：

1. 对进入 top-N 的源码候选调用 LLM triage。
2. LLM 输入必须是结构化 slice，不直接塞整段长文本。
3. LLM 输出固定 JSON：

```json
{
  "decision": "accept | reject | needs_more_context",
  "reason": "...",
  "source_to_sink": true,
  "reachable": true,
  "missing_guard": true,
  "required_context": [],
  "confidence": 0.0
}
```

4. LLM 失败、超时、JSON 不合法时，不能默认 `accept`。
5. LLM 不允许改变工具证据、验证结论或安全策略。

验收：

- 规则型源码候选必须经过 triage；
- LLM 异常不会扩大误报；
- 被拒绝候选进入 mining debug，而不是消失。

### 阶段 F：排序与预算重构

目标：让 top-N 优先保留真实可验证候选。

新的排序因子：

| 因子 | 方向 |
|---|---|
| source-to-sink 变量关联 | 强加权 |
| reachable entry | 强加权 |
| sink 危险等级 | 加权 |
| sanitizer 缺失 | 加权 |
| parser/runtime 相关性 | 加权 |
| 工具强安全规则佐证 | 加权 |
| minified/generated/test/example | 降权 |
| source/sink 缺失 | 强降权 |
| call chain 长度过短且无变量流 | 降权 |

验收：

- deep 模式增加覆盖，但不会简单放大噪声；
- top findings 中 `source_code` 的静态通过率和动态可验证率提升。

### 阶段 G：验证联动

目标：让验证阶段消费更高质量的切片。

任务：

1. 动态验证 recipe 从 `TaintSlice` 生成，而不是从泛化 finding 描述生成。
2. PoC 生成必须优先走 LLM，且必须绑定具体 source、sink、入口和 payload 形式。
3. fallback 只能生成“待补充模板”，不能冒充 PoC。
4. harness 生成基于变量级 slice，明确只证明局部链路，不标记为 `verified`。

验收：

- 不再出现 `manual-validation-payload` 或全 A payload 冒充 PoC；
- 局部动态验证报告能说明证明了哪一段链路、没有证明哪一段链路。

### 阶段 H：代码清理

目标：移除历史方案冲突和无用代码。

候选清理项：

1. `AnalysisAgent` 和 `BuiltinPatternScanner`：如果当前主流程不用，应改为 Signal provider 或删除。
2. 旧 PoC fallback：统一到 Verification/PoC 模块，禁止多个地方各自生成 payload。
3. 报告导入逻辑：区分 legacy/current schema，避免旧报告污染统计。
4. 工具可用性判断：继续收敛到 `ToolRunner`，避免不同模块各自查 PATH。
5. 类型归一化：减少 `other/environment` 直接进入 finding。

验收：

- 删除或隔离旧路径后测试通过；
- mining 主链路只有一套清晰数据模型；
- 新增模块有单元测试覆盖。

## 6. 推荐优先级

近期最应该做的顺序：

1. 阶段 A：建立评估基线；
2. 阶段 C：先过滤明显噪声；
3. 阶段 B：信号分流；
4. 阶段 D：变量级切片；
5. 阶段 E：LLM triage；
6. 阶段 F/G：排序和验证联动；
7. 阶段 H：代码清理贯穿执行。

原因是：如果没有评估基线，后续改动无法证明质量提升；如果不先过滤明显噪声，LLM 和切片都会被低价值输入拖慢；如果不做信号分流，依赖/配置/源码漏洞会继续互相污染。

## 7. 回归样例

建议固定以下项目作为质量回归集：

| 项目 | 用途 |
|---|---|
| DVWA | PHP Web 漏洞入口识别和 PoC 质量 |
| Vulnerable-Flask-App | Python source-to-sink 和 HTTP 验证 |
| maccms10 | PHP 项目、静态资产噪声过滤 |
| json-c | 小型 C 项目、parser/harness 验证 |
| Exiv2 | C++ 构建、CLI、ASAN/UBSAN 和复杂解析器链路 |
| sqlite | 中大型 C 项目、真实代码噪声控制 |

每次优化后至少记录：

- source_code finding 数量；
- verified / harness_reproduced / partial_dynamic_proof 数量；
- source 缺失率；
- sink 缺失率；
- call chain 过短比例；
- LLM triage accept/reject/needs_more_context 分布；
- top 10 finding 人工抽样判断。
