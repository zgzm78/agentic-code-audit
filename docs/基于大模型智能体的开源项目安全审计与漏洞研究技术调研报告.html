      #摘要

      摘要导读 本报告关注 LLM Agent 在开源项目安全审计、漏洞挖掘、验证与利用中的自动化边界。核心判断是：LLM 未替代传统工具，高质量研究正在转向神经-符号-动态执行的混合链路。

      项目级上下文独立验证Runtime Oracle可重放证据链

      大语言模型（Large Language Model，LLM）与智能体（Agent）技术正在改变软件安全审计和漏洞研究的自动化边界。早期研究通常将漏洞检测视为代码片段分类问题，近年来则逐步转向面向真实仓库的全项目理解、工具调用、多智能体协作、跨文件数据流推理、攻击面建模、动态验证、PoC 生成、修复与回归验证。与此同时，传统程序分析、模糊测试、符号执行、自动利用生成和运行时取证并未被 LLM 取代；相反，当前高质量研究越来越强调将 LLM 的语义理解与 CodeQL、Joern、Semgrep、AFL++、libFuzzer、KLEE、angr、Sanitizer、Nuclei 等确定性工具结合，形成“神经-符号-动态执行”混合链路。

      本文围绕课程选题一“基于大模型智能体的开源项目安全缺陷自动审计和验证系统”，系统调研以下内容：

      1. 代表性开源或公开研究系统，包括 DeepAudit、AgentStalker、OpenSecurity、ESAA-Security、Sandyaa、Vulnhuntr、OpenAnt、Visa VVAH、GitHub Security Lab Taskflow Agent、RepoAudit、RAPTOR、PentestGPT 等；
      2. 经典与现代漏洞挖掘工具，包括 CodeQL、Semgrep、Joern、AFL++、libFuzzer、syzkaller、boofuzz、WinAFL、KLEE、angr、Triton、Ghidra、Qiling 等；
      3. 漏洞验证、PoC 和利用相关工具，包括 Nuclei、OWASP ZAP、sqlmap、Commix、Metasploit、pwntools、rex、Mechanical Phish 等；
      4. 运行时验证、沙箱和证据固化技术，包括 ASan、UBSan、CASR、rr、Tracee、nsjail 和 gVisor；
      5. 代表性论文、工业界实践与 2025-2026 年最新研究动态；
      6. CyberGym、CyberGym-E2E、CVE-Bench、ExploitGym、SEC-bench Pro、CWE-Bench-Java、Vul4J、Magma、NIST SARD/Juliet 等评测基准；
      7. 当前仍未解决的核心问题和具有研究价值的创新方向。

      调研表明，当前研究前沿已经从“LLM 能否发现漏洞”转向“如何构建可靠、可验证、可复现的自动化漏洞研究 Harness”。项目级上下文选择、程序图与 LLM 的结合、发现与验证分离、对抗式验证、确定性运行时 Oracle、可利用性分级、差分验证、自动生成 Fuzz Harness、闭环覆盖反馈和可重放证据链，是目前最值得关注的研究方向。

      #2.1 传统阶段：规则、查询与程序分析

      传统自动化代码审计主要依赖：

      • 语法模式匹配；
      • AST 规则；
      • 控制流分析；
      • 数据流与污点分析；
      • 符号执行；
      • 模糊测试；
      • 动态应用安全测试。

      其典型工具包括 Semgrep、CodeQL、Joern、KLEE、AFL++、libFuzzer 等。传统方法的主要优势是：

      1. 结果可解释；
      2. 执行行为相对确定；
      3. 易于复现；
      4. 在已知漏洞模式上可规模化部署。

      主要局限则包括：

      • 项目特定 source、sink 和 sanitizer 难以建模；
      • 业务逻辑漏洞难以用统一规则描述；
      • 动态框架、反射、回调和复杂依赖会降低静态精度；
      • Fuzzing 容易受 Harness、输入格式和深层状态限制；
      • 符号执行容易遭遇路径爆炸。

      #2.2 LLM 阶段：从代码分类到项目语义理解

      早期 LLM 漏洞检测常见流程是：

      代码片段
  ↓
Prompt
  ↓
Vulnerable / Safe

      这种方法对真实项目存在明显问题：

      • 缺少跨文件上下文；
      • 难以验证调用路径；
      • 容易把危险 API 的出现误判为真实漏洞；
      • 无法证明攻击者可控性与实际影响；
      • 结果高度依赖 Prompt 和模型版本。

      因此，近年的研究开始转向：

      Repository
  ↓
Structure / Graph / Threat Model
  ↓
Targeted Context
  ↓
LLM Reasoning
  ↓
Tool-assisted Verification

      #2.3 Agent 阶段：从一次性回答到迭代式漏洞研究

      Google Project Zero 的 Project Naptime 强调模拟人工漏洞研究员的工作方式：形成假设、查看代码、运行工具、观察结果、修改假设、继续调查。后续 Big Sleep 报告了一个在 SQLite 中发现并在正式版本发布前修复的真实可利用问题。这类工作说明，智能体的价值不只是“多轮聊天”，而是可持续的假设-工具-证据循环。

      #2.4 2025-2026：Harness 成为核心研究对象

      到 2026 年，公开研究和工业实践出现共同趋势：

      • Visa VVAH：从仓库调查、威胁建模、分析规划到验证、报告、修复和再验证；
      • OpenAnt：可达性裁剪、发现者与攻击者对抗、沙箱验证；
      • Cloudflare：Discovery Harness 与 Validation System 分离；
      • RAPTOR：把静态分析、二进制分析、Fuzzing、验证、Exploit 和 Patch 编排到一个工作流；
      • GitHub Security Lab Taskflow Agent：使用声明式任务流与 MCP 工具组织安全研究；
      • DARPA AIxCC：把自动发现与自动修复放入真实开源软件竞赛环境。

      因此，当前研究对象已经逐渐从单一模型转为：

      结论

      模型 + 上下文选择 + 工具 + Agent 角色 + 状态管理 + 动态验证 + 证据与评测。

      #3.1 DeepAudit

      #3.1.1 项目定位

      DeepAudit 是课件明确列出的参考系统，也是目前公开工程完成度相对较高的多智能体代码审计项目之一。其公开架构通常包含：

      Orchestrator
    ↓
Recon / Repository Understanding
    ↓
Analysis / Audit Agent
    ↓
Verification Agent
    ↓
Sandbox PoC
    ↓
Report

      其主要技术特点包括：

      • 多智能体协作；
      • 仓库级代码理解；
      • AST/RAG 等上下文能力；
      • 自动生成 PoC；
      • Docker 沙箱验证；
      • 结构化漏洞报告。

      #3.1.2 研究价值

      DeepAudit 最值得借鉴的不是“用了多个 Agent”，而是尝试将代码扫描、复核、PoC 和报告连接为闭环。这与课程题目要求高度一致。

      #3.1.3 局限与注意事项

      • 多智能体本身并不自动保证独立性；如果发现者与验证者使用相同模型、相同上下文和高度相似 Prompt，可能产生相关性幻觉；
      • 自动生成 PoC 与真实漏洞确认之间仍需确定性运行时证据；
      • 项目公开的漏洞成果应逐项通过 CVE/NVD/厂商公告核验。

      资料：DeepAudit GitHub★ 6.6k

      #3.2 AgentStalker

      #3.2.1 项目定位

      AgentStalker 强调把 LLM Agent 或软件系统作为整体对象进行端到端安全审计，其公开方法包含：

      Static Modeling
      ↓
Attack Graph Synthesis
      ↓
Sandbox Dynamic Verification
      ↓
Evidence Adjudication

      其设计涉及：

      • typed taint graph；
      • 攻击图合成；
      • Docker 沙箱；
      • 网络服务模拟；
      • eBPF/Tracee 行为观测；
      • OPA 策略；
      • 证据裁决。

      #3.2.2 研究价值

      AgentStalker 的重要贡献是把“漏洞发现”升级为“系统建模-攻击路径-动态证据-裁决”。这种思想比单纯输出漏洞清单更符合可验证安全研究。

      #3.2.3 局限

      公开资料也提示其仍存在：

      • 部分污点能力以过程内分析为主；
      • MCP 等生态适配存在限制；
      • 缺少大规模、独立、公开的项目级实证评估。

      因此，更适合把它视为方法论和研究框架，而不是已经充分验证的通用工业扫描器。

      资料：AgentStalker GitHub★ 128

      #3.3 OpenSecurity

      OpenSecurity 更接近 AI 驱动的多领域安全分析平台，而非单一代码漏洞扫描器。其规划通常覆盖：

      • Binary Agent；
      • Mobile Agent；
      • Web Agent；
      • AI Security Agent；
      • Coordinator。

      其核心启示是三层分工：

      AI Orchestration
       ↓
Deterministic Security Tools
       ↓
Knowledge / Evidence

      这种架构强调：LLM 适合做工具编排、假设形成和语义解释，而 AST、反汇编、数据流、文件哈希、执行结果等事实应由确定性工具产生。

      资料：OpenSecurity GitHub★ 34

      #3.4 ESAA-Security

      #3.4.1 核心思想

      ESAA-Security 的代表性思想是 Event-Sourced Agent Architecture，即事件溯源式智能体架构。它不把一次安全审计视作不可解释的对话，而是将每个阶段记录为事件：

      RepositoryParsed
CandidateFound
PathConfirmed
VerifierStarted
PoCGenerated
SandboxExecuted
EvidenceCaptured
VerdictIssued

      事件日志具有：

      • append-only；
      • 可校验；
      • 可回放；
      • 可重建状态；
      • 可追踪输入和输出。

      #3.4.2 研究意义

      这一路线直接回应了 LLM 安全审计的核心缺陷：

      • 结果不可复现；
      • 中间推理不可审计；
      • 模型版本变化导致结论漂移；
      • 失败难以定位。

      #4.1 程序表示与静态分析

      #4.1.1 Code Property Graph：代码属性图

      2014 年 IEEE Symposium on Security and Privacy 的经典论文 Modeling and Discovering Vulnerabilities with Code Property Graphs 提出了 Code Property Graph（CPG）。其中，CFG 表示控制流图，PDG 表示程序依赖图。其基本思想是将：

      • AST（Abstract Syntax Tree）；
      • CFG（Control Flow Graph）；
      • PDG（Program Dependence Graph）

      融合为统一属性图，并使用图遍历表达漏洞模式。

      CPG 的意义在于，它把原本分散的程序事实统一起来：

      Function A
  ├── AST → Syntax Node
  ├── CFG → Branch / Loop
  ├── Call → Function B
  ├── Data Flow → Variable C
  └── Dependency → Sink D

      对大模型而言，CPG 能够提供比原始代码更紧凑、更结构化的上下文。当前“Graph + LLM”“CPG + MCP”“程序切片 + LLM”研究均可追溯到这一基础。

      经典论文：Yamaguchi et al., IEEE S&P 2014IEEE S&P 2014

      • 与 MCP/Agent 结合潜力高。

      #局限

      • 不同语言前端成熟度存在差异；
      • 大型项目建图和查询有成本；
      • 框架动态行为和运行时反射仍可能超出静态图表示。

      资料：Joern Documentation；CPG Documentation

      #4.1.3 CodeQL

      CodeQL 将代码构建为数据库，并通过 QL 查询语言进行语义分析，是现代静态安全分析的重要代表。

      #主要能力

      • 控制流分析；
      • 局部与全局数据流；
      • Taint Tracking；
      • 跨函数路径查询；
      • Variant Analysis；
      • SARIF 输出。

      典型安全分析：

      Source
  ↓
Propagation
  ↓
Sanitizer?
  ↓
Sink

      #对 LLM 研究的价值

      CodeQL 可以承担三种角色：

      1. 候选生成器：产生高召回的安全告警；
      2. 路径事实提供者：给出 source-to-sink 传播路径；
      3. 规则执行后端：由 LLM 生成或补充项目特定 source/sink 规则。

      2023 年 GitHub CodeQL 团队就公开介绍过使用 LLM 辅助 API 建模，以扩展 sink 等语义模型。这是“LLM 不替代静态分析，而是补充其语义规格”的重要实践。

      #局限

      • 对动态语言、反射和复杂框架的精度依赖模型库；
      • 查询编写门槛较高；
      • 全局数据流具有计算成本；
      • source 到 sink 的路径存在并不自动等同于可利用漏洞。

      资料：CodeQL Data Flow Documentation；GitHub Security Lab: LLM-assisted CodeQL modeling

      #4.1.4 Semgrep

      Semgrep 是高效、易扩展的代码模式与静态分析工具。

      #适用能力

      • 危险 API 检测；
      • 代码模式扫描；
      • 配置缺陷；
      • 硬编码密钥模式；
      • taint mode；
      • 项目自定义规则。

      Semgrep 的优势是规则开发快、语言覆盖广、适合批量扫描和 CI/CD。

      #与深层程序分析的分工

      更合理的定位是：

      Semgrep
  → 快速发现候选、局部模式、危险调用

CodeQL / Joern
  → 深层跨过程路径、调用关系、程序切片

      不应把 Semgrep Community Edition 简单描述为完整的跨文件程序分析器；高级跨函数、跨文件能力与具体产品/引擎配置有关。

      资料：Semgrep Taint Analysis

      #4.1.6 DongTai IAST工具

      #4.2 模糊测试（Fuzzing）

      Fuzzing 是未知漏洞挖掘的核心技术之一。与 LLM 结合后，研究焦点逐渐从“让 LLM 代替 Fuzzer”转向“让 LLM 改善 Harness、字典、输入结构、目标选择和覆盖率反馈”。

      #4.2.1 AFL++

      AFL++ 是现代覆盖率引导模糊测试的重要开源平台。

      #主要能力

      • 编译插桩；
      • QEMU 模式；
      • Frida 模式；
      • Unicorn 模式；
      • Persistent Mode；
      • CmpLog；
      • 自定义 Mutator；
      • 多实例并行。

      #适用对象

      • C/C++ 程序；
      • Parser；
      • 文件格式处理；
      • 协议实现；
      • 二进制程序。

      #与 LLM Agent 的结合方向

      High-risk Function Discovery
          ↓
LLM Generates Fuzz Harness
          ↓
Compile
   ↓ fail        ↓ success
Repair Agent      Fuzz
                    ↓
               Coverage / Crash
                    ↓
             Harness Refinement

      真正有研究价值的是闭环，而不是“把 AFL++ 接进系统”。

      资料：AFL++ Documentation

      #4.2.2 libFuzzer

      libFuzzer 是 LLVM 生态中的进程内、覆盖率引导、进化式 Fuzzer。

      典型接口：

      int LLVMFuzzerTestOneInput(const uint8_t *Data, size_t Size) {
    // target API
    return 0;

      • 库函数；
      • Parser；
      • Serializer；
      • Codec；
      • C/C++ 核心 API。

      libFuzzer 与 SanitizerCoverage 和 ASan 等工具结合紧密，非常适合作为“LLM 自动生成 Harness”的实验后端。

      资料：LLVM libFuzzer Documentation

      #4.2.3 syzkaller

      syzkaller 是无监督、覆盖率引导的内核 Fuzzer，最初面向 Linux，目前公开支持或扩展到多个内核和系统目标，是内核漏洞挖掘方向最具代表性的工具之一。

      #核心特点

      • 系统调用描述；
      • 程序生成与变异；
      • 覆盖率反馈；
      • VM/设备执行；
      • 崩溃收集与最小化；
      • 与 KASAN/KMSAN/KUBSAN 等内核检测器结合。

      #为什么与 LLM 研究有关

      内核 Fuzzing 的瓶颈之一是 syscall specification。KernelGPT 等研究已经探索使用 LLM 自动生成 syzkaller 系统调用规格。这说明 LLM 最适合填补“人工语义建模成本”，而不是替代覆盖率引导执行。

      资料：syzkaller GitHub★ 6.3k

      #4.2.4 boofuzz

      boofuzz 是 Sulley 的继任者之一，适合网络协议和状态化交互 Fuzzing。

      #主要特点

      • 定义协议消息结构；
      • 字段级变异；
      • Session 管理；
      • 请求序列；
      • 目标连接；
      • Crash 监控。

      #适用场景

      • 自定义 TCP/UDP 协议；
      • 工控协议；
      • 网络设备服务；
      • 状态机驱动服务。

      #与 LLM 结合的潜力

      LLM 可辅助：

      • 从协议文档生成消息模型；
      • 从源码推断状态序列；
      • 生成高价值字段字典；
      • 对 Crash 和响应差异进行语义聚类。

      资料：boofuzz Documentation

      #4.2.5 WinAFL

      WinAFL 是针对 Windows 程序的 AFL 风格 Fuzzer。

      #适用对象

      • Windows 闭源程序；
      • DLL；
      • 文件解析器；
      • 驱动或桌面软件中的用户态组件。

      #研究价值

      #4.3 符号执行与约束求解

      #4.3.1 KLEE

      KLEE 是 2008 年 OSDI 的经典符号执行系统。

      普通执行只测试具体值：

      x = 3

      符号执行把输入视为：

      x = Symbolic

      并为不同分支维护路径约束，例如：

      Path 1: x > 100
Path 2: x <= 100

      求解器可自动构造满足路径的输入。

      #研究意义

      KLEE 证明了自动路径探索和输入生成的可行性，也是后续 AEG、Concolic Execution 和混合 Fuzzing 的基础之一。

      #局限

      • 路径爆炸；
      • 环境建模；
      • 系统调用与外部依赖；
      • 大型真实程序扩展性。

      #与 LLM 的新关系

      LLM 可以：

      • 选择高价值路径；
      • 推断约束目标；
      • 解释复杂环境；
      • 生成驱动代码。

      而求解器负责确定性求值。

      资料：KLEE Project；KLEEOSDI 2008

      #4.3.2 angr

      angr 是面向二进制分析的开源框架，支持：

      • Binary Lifting；
      • CFG 恢复；
      • 符号执行；
      • 约束求解；
      • Reachability；
      • 数据流与状态探索。

      典型任务：

      Target Address
      ↓
Symbolic State Exploration
      ↓
Path Constraints
      ↓
Concrete Input

      适合：

      • 无源码程序；
      • Crash 分析；
      • 二进制验证；
      • 自动利用研究。

      资料：angr

      #4.3.3 Triton

      Triton 是动态二进制分析框架，支持：

      • 动态符号执行；
      • Taint Analysis；
      • SMT 接口；
      • 指令级语义；

      与 angr 相比，Triton 更接近可嵌入的动态符号执行基础设施。

      资料：JonathanSalwan/Triton★ 4.2k

      #6. 漏洞验证、PoC 与利用工具

      本章导读 本章区分验证、PoC 和 Exploit 的层级。课程实现中应避免让同一个发现 Agent 自我确认，而应把候选发现交给独立验证链路。

      NucleiZAPsqlmapMetasploitpwntools
      验证链路

      原文强调 Nuclei、ZAP、sqlmap、Commix、Metasploit、pwntools 等工具任务不同，不能混为一谈。

      1. 候选发现
      2. 路径事实
      3. 语义分析
      4. 独立验证
      5. 专项动态工具
      6. 确定性 Oracle
      7. 证据与回放

      漏洞验证工具需要根据功能区分为三类：

      1. 已知漏洞模板验证器：如 Nuclei；
      2. 漏洞类型专项验证器：如 sqlmap、Commix；
      3. 利用开发与自动利用基础设施：如 Metasploit、pwntools、rex。

      这三类工具不能混为一谈。

      #6.1 Nuclei

      Nuclei 是模板驱动的安全验证工具，其核心结构是：

      Request
  +
Matcher
  +
Extractor

      模板可以描述：

      • HTTP 请求；
      • 网络协议；
      • DNS；
      • 文件检查；
      • Headless 流程；
      • 工作流。

      #研究价值

      相比让 LLM 直接生成任意 Python PoC，让模型先生成结构化验证规格再转换为 Nuclei Template 更容易：

      • 校验；
      • 审计；
      • 重放；
      • 版本管理。

      #局限

      Nuclei 更擅长：

      • 已知漏洞；
      • 可模板化的 Web/网络行为；

      而不擅长自动理解任意源码中的新型复杂业务逻辑漏洞。

      资料：projectdiscovery/nuclei★ 29.5k；nuclei-templates★ 12.6k

      #6.2 OWASP ZAP

      OWASP ZAP 是漏洞研究工具链中不可缺少的开源 DAST 工具。

      其功能包括：

      • 代理；
      • 被动扫描；
      • 主动扫描；
      • Spider；
      • API 自动化；
      • 手工安全测试辅助。

      #与 Nuclei 的差异

      Nuclei
→ 高度模板化、快速验证、社区模板生态

ZAP
→ 完整 Web DAST、代理、爬虫、主动/被动扫描

      ZAP 更适合对运行中的 Web 项目做广泛动态测试；Nuclei 更适合作为结构化、可复现的专项验证器。

      资料：OWASP ZAP；OWASP WSTG Tool Resource

      #8.15 Revelio：成本优化的 Agentic 内存安全漏洞检测（2026）

      #研究问题

      内存安全漏洞即便在经过大规模 fuzz 和人工审计的成熟项目中仍持续存在。直接使用大模型进行仓库级漏洞检测面临三方面挑战：

      • 幻觉与不可靠
      • 成本随代码规模线性甚至超线性增长
      • 结果缺乏可验证证据。

      #方法

      Revelio 提出一种代价可控的 Agentic 检测框架，核心设计包括：

      1. 幻觉抑制：Agent 不直接输出漏洞判断，而是生成可执行的 Proof-of-Vulnerability 载荷，通过 ASan 等确定性 Sanitizer 复现确认；
      2. 成本优化：使用廉价 LLM 配合轻量静态分析形成漏洞候选并优先级排序，仅将高价值候选交给昂贵模型；
      3. 验证管道：漏洞只有在 Sanitizer 复现后才进入报告，未复现结果不上榜。

      #论文报告的实证

      • 在 7 个经过持续 5–8 年 fuzz 的高质量项目上发现 19 个未知内存安全漏洞
      • 在 CyberGym 的 100 个 Arvo 项目上完成大规模评测
      • 总花费约 $300，每项目平均约 1 小时
      • 在同等 Token 预算下优于前沿 coding agent。

      #研究意义

      Revelio 是 2026 年最直接印证本报告核心结论的实证工作之一：

      • 它同时兑现了"神经-符号-动态混合"（第 11.1 节）与"确定性 Runtime Oracle"（第 11.7 节）两条路线
      • 它以量化数据证明"廉价 LLM + 严格 Sanitizer 验证"在成本与召回率上均优于"昂贵 LLM 单打独斗"
      • 它给"验证正在取代发现成为主要难点"（第 12.5、14.8 节）提供了正向工程证据。

      #研究背景与目标域说明

      Make Agent Defeat Agent: Automatic Detection of Taint-Style Vulnerabilities in LLM-based Agents（Fengyu Liu、Yuan Zhang、Hao Chen 等，复旦大学 + UC Davis）发表于 USENIX Security 2025。

      目标域说明：论文的直接检测对象是 LLM Agent 本身（污点从恶意 prompt 流向敏感操作），而非通用开源应用代码。因此它更贴近选题二的问题域。将其纳入本调研的原因在于其方法论对选题一的验证智能体设计具有直接可迁移性。

      #方法与迁移价值

      论文的核心方法是"让一个 Agent 主动挑战另一个 Agent 的假设"：

      Finder Agent 提出污点候选
        ↓
Attacker Agent 主动构造反例
        ↓
沙箱验证
        ↓
经受住反例的发现才进入报告

      这与本调研第 3.7 节 OpenAnt 的 Defender/Attacker 对抗验证、第 13.4 节的对抗式独立验证方向属于同一研究谱系。由于本文是 USENIX Security 顶会论文，为该方法论提供了明确的学术背书，可作为选题一验证智能体设计的可引用依据。

      #研究意义

      即便目标域不同，其研究结论仍支持本报告的两个判断：

      1. 多 Agent 系统的价值不在数量，而在角色对抗性（第 14.1 节）；
      2. 同模型、同上下文的多 Agent 复核容易产生相关性幻觉（第 12.3 节），必须让某个 Agent 主动扮演反方。

      #12.1 大型仓库上下文爆炸

      #问题

      直接把仓库文件全部输入模型会导致：

      • Token 成本；
      • 无关上下文干扰；
      • 调用路径丢失；
      • 关键安全边界被淹没。

      #尚未完全解决

      现有方法包括：

      • RAG；
      • 调用图；
      • CPG；
      • 可达性；
      • 程序切片。

      但不同语言、框架和动态特性使“最佳上下文选择”仍是开放问题。

      #12.2 危险代码不等于漏洞

      例如：

      os.system(cmd)

      仅看到 sink 不足以证明 RCE。

      必须回答：

      1. cmd 是否攻击者可控？
      2. 调用路径是否可达？
      3. 是否经过净化？
      4. 是否需要管理员权限？
      5. 默认配置是否启用？
      6. 实际能否产生安全影响？

      当前很多 LLM 检测器仍会在这里产生误报。

      #12.3 相关性幻觉

      多个 Agent 使用相同：

      • 模型；
      • 上下文；
      • Prompt 风格；

      可能重复相同错误。

      因此“3 个 Agent 投票”不一定是真正独立验证。

      #12.4 验证 Agent 自身也会误判

      Sifting the Noise 证明 Agent 可以显著降低 SAST 噪声，但也指出激进过滤可能删除真实漏洞。

      因此验证器需要：

      • 自己的 Recall/Precision；
      • CWE 分层结果；
      • 成本；
      • 误删率。

      不能把 Verifier 视作神谕。

      #12.5 PoC 生成仍是瓶颈

      CyberGym 等 benchmark 表明：

      • 定位漏洞；
      • 理解触发条件；
      • 配置环境；
      • 构造输入；
      • 判断副作用；

      是不同难题。

      因此：

      Finding ≠ PoC
PoC ≠ Exploit
Exploit ≠ Reproducible Evidence

      #12.6 自动 Exploit 的任务定义不清晰

      Web 路径遍历与内存破坏漏洞的“利用”完全不同：

      • 文件读取：证明读到随机 Canary 即可；
      • SSRF：证明内部 Mock Service 收到请求；
      • SQLi：证明查询语义被控制；
      • 内存漏洞：Crash 只是起点，可能还要证明控制能力。

      因此通用 Exploit Success 指标不合理。

      #12.7 结果不可复现

      Agent 运行受：

      • 模型更新；
      • 温度；
      • 工具版本；
      • 仓库 commit；
      • 依赖；
      • 环境；

      影响。

      没有事件日志、Artifact Hash、容器镜像和 Replay 命令，就很难称为可复现研究。

      #12.8 Benchmark 与真实研究存在差距

      Synthetic benchmark 容易；真实 0-day Hunting 难。

      Known CVE PoC 生成容易泄露先验；Blind Discovery 更难。

      因此需要同时使用：

      • 合成基础测试；
      • 已知真实漏洞；
      • 隐藏漏洞位置；
      • 活跃开源项目；
      • 动态 Oracle。

      #12.9 成本与可扩展性

      项目级研究显示 LLM 漏洞检测可能消耗非常高的 Token 和时间。

      研究不能只报告 Accuracy，还应报告：

      • Token / Repository；
      • Cost / Finding；
      • Time / Verified Vulnerability；
      • 工具调用数；
      • Context Reduction Ratio。

      如何自动识别：

      • 外部入口；
      • 信任边界；
      • 可达函数；
      • 高风险 Sink；
      • 安全检查；

      并构造最小但充分的安全上下文？

      • Route/Handler Discovery；
      • 调用图可达性；
      • CPG Program Slicing；
      • Data Dependency；
      • Threat-model-guided Prioritization；
      • Dynamic Trace 补充静态图。

      #潜在创新

      不是简单“减少 Token”，而是建立：

      结论

      漏洞相关上下文完整性与上下文规模之间的可测量权衡。

      • Context Reduction Ratio；
      • Vulnerability Recall；

      #13.2 LLM 自动推断项目特定 Source、Sink 与 Sanitizer

      #背景

      #研究问题

      如何识别框架或项目自定义函数：

      get_user_payload() → Source?

      • 用多视角 Agent 生成候选；
      • 让 CodeQL/Joern 反向验证规格是否产生合理路径。

      #关键问题

      LLM 生成的 Source/Sink 不能直接作为事实，需要：

      • 规则编译；
      • 实际查询；
      • 人工或动态抽查；

      #13.4 对抗式独立验证

      #背景

      OpenAnt、VVAH、Cloudflare 都强调发现与验证分离，Anthropic Defending Code Reference Harness 进一步以定量数据佐证独立 Verifier + 强制 PoC 可将误报率压至接近 0，Make Agent Defeat Agent（USENIX Security 2025）在 LLM Agent 污点检测场景给出了该方法论的顶会实证。

      #研究问题

      如何避免 Verifier 只是重复 Finder 的偏见？

      #可能方法

      • 不同模型
      • 不同 Prompt
      • 不同上下文
      • Finder 只提交结构化 Claim
      • Verifier 不看 Finder 的长篇自然语言推理
      • Verifier 的目标是寻找反例。

      #Finding Contract 示例

      {
  "claim": "attacker-controlled input reaches shell execution",
  "cwe": "CWE-78",
  "entry_point": "...",
  "source": "...",
  "sink": "...",
  "call_path": [],
  "preconditions": [],
  "expected_impact": "...",
  "falsification_conditions": []
}

      #研究价值

      可以专门研究：

      • 同模型 vs 异模型
      • 完整推理共享 vs Claim-only
      • 普通复核 vs 对抗证伪。

      #13.5 可利用性分级而非二元分类

      #背景

      ExploitGym 和 ExploitBench 等研究说明，漏洞存在与成功利用之间存在多个能力层级。

      #一种可研究的分级方式

      #V0：Candidate

      存在危险模式或可疑语义。

      #V1：Reachable

      攻击入口能够到达安全敏感操作。

      #V2：Attacker-Controlled

      攻击者输入真实影响关键操作。

      #V3：Triggered

      构造输入能够触发异常或目标行为。

      #V4：Impact-Confirmed

      确定性证据证明真实安全影响。

      #V5：Differentially Reproducible

      漏洞版本稳定成功，修复版本稳定失败，并可多次重放。

      #研究意义

      这种分级比：

      Vulnerable = true

      更能描述证据强度。

      #可研究问题

      • 不同漏洞类型是否需要不同等级？
      • V2 到 V3 的转化率是多少？
      • 哪些 SAST 告警能达到 V4？
      • Agent 在哪一层失败最多？

      #13.6 漏洞类型专用的确定性 Runtime Oracle

      #背景

      LLM 自我判断 PoC 成功不可靠。

      #SQL Injection Oracle

      可以使用：

      • True/False Payload 差分
      • DB 查询日志
      • 行数差异
      • 时间差异（需控制噪声）。

      #Command Injection Oracle

      使用随机 Canary Token，验证受控副作用，例如：

      • 特定文件创建
      • Mock 命令调用
      • 特定子进程事件。

      #Path Traversal Oracle

      运行时随机生成：

      /secret/canary-<uuid>

      只有越权读取成功才能获得该值。

      #SSRF Oracle

      内部 Mock Service 生成唯一 nonce，只有目标程序主动请求才判定成功。

      #Auth Bypass Oracle

      对比：

      • 未认证身份
      • 低权限身份
      • 高权限身份

      访问受保护资源的结果。

      #Memory Safety Oracle

      • ASan/UBSan
      • Crash Stack
      • Input Hash
      • Repeat Replay。

      Revelio（第 8.15 节）在 7 个持续 fuzz 5–8 年的成熟项目上以 ASan 复现为唯一判定条件，用约 $300 总花费发现 19 个未知内存安全漏洞，是该 Oracle 设计路径在 2026 年最直接的工程实证。

      #创新价值

      建立漏洞类别—验证器—Oracle映射是当前自动验证系统的重要研究空间。

      #13.7 差分漏洞验证

      #基本思想

      使用同一个 PoC：

      Vulnerable Commit
PoC = Success

Fixed Commit
PoC = Fail

      #价值

      差分验证可以减少：

      • 环境噪声；
      • 偶然 Crash；
      • 错误响应判断；
      • Agent 主观判断。

      #可扩展方向

      • Patch 前后动态 Trace 差分；
      • Source/Sink Path 差分；
      • 覆盖率差分；
      • 权限差分。

      Vul4J、CVE 修复提交等数据非常适合该研究。

      #13.8 可重放证据链与 Event-Sourced Audit

      #背景

      ESAA-Security 和 Cloudflare 的状态管理实践说明，Agent 长任务需要持久化。

      RepositoryCheckedOut

      #每个事件可包含

      • repository commit；
      • model/version；
      • prompt hash；
      • tool/version；
      • input artifact hash；
      • output artifact hash；
      • parent event；
      • previous hash。

      #研究问题

      • 哪些 Agent 决策能够确定性重放？
      • 模型变化后如何复验旧 Finding？
      • 证据链完整性如何量化？

      #背景

      • AI-SBOM；

      • 版本与配置判断；
      • LLM 辅助库语义映射。

      #13.12 自动生成安全 Threat Model

      #背景

      VVAH 和 Codex Security 都强调项目特定威胁模型。

      #研究问题

      如何自动从代码提取：

      • Assets；
      • Trust Boundaries；
      • Actors；
      • Entry Points；
      • Privileged Operations；
      • Likely Threat Classes。

      #风险

      自动 Threat Model 也可能遗漏关键资产或错误判断信任边界。

      #可评测方法

      • 与人工 Threat Model 对比；
      • 对漏洞 Recall 的贡献；
      • 是否减少无关扫描；
      • Token/Cost 变化。

      #13.13 专项安全工作流（Taskflow）

      通用 Prompt 很难同时覆盖：

      • SQLi；
      • Auth Bypass；
      • SSRF；
      • Memory Corruption；
      • Race Condition。

      因此可研究：

      Vulnerability Class
      ↓
Specialized Workflow
      ↓
Specialized Evidence Contract

      例如：

      • SQLi 工作流强调 query construction 和 DB Oracle；
      • Path Traversal 强调 canonicalization；
      • Auth Bypass 强调 identity/role differential；
      • Memory Bug 强调 Sanitizer 与 Crash Control。

      GitHub Taskflow Agent 是这一方向的重要参考。

      #13.14 多模型异构协作

      当前多 Agent 往往只是同一个模型的多个角色。

      更值得研究的是：

      • 强模型做语义发现；
      • 轻量模型做信息抽取；
      • 不同厂商模型做独立验证；
      • 专项模型做代码或二进制分析。

      #研究问题

      • 异构模型是否降低相关性错误？
      • 成本是否可控？
      • 哪些任务值得使用强模型？
      • 哪些阶段可以由规则或脚本替代？

      #13.15 验证器的自我评测与置信校准

      如果 Verifier 会误删真实漏洞，则需要评估：

      • True Positive Retention；
      • False Positive Rejection；
      • CWE-specific Performance；
      • Calibration；
      • Abstention。

      可以允许 Verifier 输出：

      Confirmed
Rejected
Insufficient Evidence

      而不是强制二元判断。

      #13.16 从漏洞报告生成到科学证据对象

      传统报告字段：

      • Title；
      • Severity；
      • Description。

      更适合自动化漏洞研究的 Evidence Object 可以包含：

      • Repository Commit；
      • File/Line；
      • Entry Point；
      • Source；
      • Sink；
      • Call/Data-flow Path；
      • Preconditions；
      • Payload；
      • Runtime Logs；
      • Oracle Result；
      • Replay Command；
      • Fixed-version Result。

      这使漏洞 Finding 从文本结论转为可计算、可比较的研究对象。

      #14. 综合批判性综述

      本章导读 本章给出对领域常见叙事的校正：多 Agent、最新模型和工具堆叠本身不是创新，真正要评价的是证据质量和可复现验证。

      方法创新证据质量SAST 互补验证瓶颈

      #14.1 “多 Agent”本身已经不是创新

      目前已有大量系统采用：

      Recon Agent
Audit Agent
Verify Agent
Report Agent

      因此仅增加 Agent 数量不能证明研究价值。

      真正有意义的问题是：

      • 角色是否独立？
      • 工具事实如何传递？
      • 状态如何持久化？
      • 失败如何恢复？
      • 验证如何证伪？
      • 结果如何重放？

      #14.2 “使用最新模型”不是方法创新

      模型更新很快。如果研究贡献只是：

      换成更强模型 → 指标提高

      则结果难以持续。

      更稳定的贡献是：

      • Context Selection；
      • Program Representation；
      • Verification Protocol；
      • Runtime Oracle；
      • Evidence Model；
      • Benchmark Methodology。

      #14.3 SAST 与 LLM 不是替代关系

      项目级实证研究表明：

      • LLM 有独特发现；
      • SAST 也有独特发现；
      • 两者都可能有高误报；
      • LLM 成本高；
      • SAST 语义建模难。

      更可信的研究方向是：

      SAST / CPG / Fuzzing
          +
LLM Semantics / Planning
          +
Dynamic Verification

      #14.4 发现能力与证据质量必须分开评价

      一个系统可能：

      • Recall 很高；
      • 但全部是 V0/V1 候选。

      另一个系统可能：

      • 发现少；
      • 但大部分达到 V4/V5。

      因此应该分别评价：

      • Discovery Recall；
      • Verification Precision；
      • Impact Confirmation；
      • Replayability。

      #14.5 自动 PoC 与自动 Exploit 应严格区分

      #PoC

      证明漏洞现象或安全影响。

      #Exploit

      获得特定攻击能力，例如：

      • 任意代码执行；
      • 任意读写；
      • 权限提升；
      • 身份绕过。

      很多系统把“构造触发请求”称为 Exploit，但在内存安全研究中这可能只达到 Crash/PoV。

      报告中应准确使用术语。

      #14.6 工程成熟度与学术证据强度应分开

      例如：

      • 某开源项目 UI 完整、功能丰富，但缺少 benchmark；
      • 某论文实验严谨，但代码原型很难部署。

      因此比较系统时最好设置两个维度：

      #Engineering Maturity

      • 安装；
      • 文档；
      • 多语言；
      • 稳定性；
      • UI/API。

      #Evidence Strength

      • Ground Truth；
      • Baseline；
      • Ablation；
      • Independent Verification；
      • Reproducibility。

      不要把 GitHub Star 数量当成学术证据。

      #14.7 2026 年最新结果需要防止“宣传数字误读”

      OpenAI、Anthropic、Cloudflare 等机构公开了大量：

      • 扫描规模；
      • 漏洞数量；
      • 修复数量；
      • 噪声下降数据。

      这些数据具有重要工程参考价值，但通常：

      • 数据集不可完全公开；
      • 模型和系统可能闭源；
      • 人工筛选流程不透明；
      • 不能直接与学术 benchmark 横向比较。

      因此报告应写作：

      证据口径

      “官方报告称……”

      而不是：

      结论

      “该系统已经证明达到……”

      #14.8 当前最薄弱的环节是验证，不是发现

      现代 LLM 很容易生成大量“看起来合理”的 Finding。

      困难在于证明：

      Attacker Control
   +
Reachability
   +
No Effective Defense
   +
Runtime Impact

      因此漏洞验证、Oracle、差分实验和证据回放比增加一个扫描 Agent 更具有研究价值。

      Anthropic Defending Code Reference Harness 的公开博客对该判断给出了同方向的工程佐证：截至 2026 年 5 月 22 日，Anthropic 自扫累计披露 1,596 个漏洞，但仅有 97 个完成修补——"发现的产能远超验证与修补的产能" 已经不是理论推测，而是被大厂运营数据直接印证的产业现实。

      #14.9 真实研究将越来越依赖 Harness

      未来高水平系统的差异可能更多来自：

      • 代码检索方式；
      • 工具接口；
      • Agent 拓扑；
      • 数据持久化；
      • 动态环境；
      • 验证协议；
      • 反馈机制。

      而不是单一模型排行榜。

      从技术关系看，比较自然的研究链路是：

      候选发现
    ↓

      #16. 结论

      本章导读 结论部分把全文收束为一个判断：LLM Agent 的研究价值正在从“直接给答案”转向“组织可验证的漏洞研究过程”。

      混合链路独立验证可复现证据

      本次调研覆盖了大模型智能体代码审计、传统程序分析、模糊测试、符号执行、漏洞验证、PoC/Exploit、运行时取证、沙箱和评测基准等多个方向。

      综合来看，可以得出以下结论。

      第一，LLM 并没有替代传统漏洞研究技术。 当前最具说服力的研究普遍采用混合方法：程序分析提供结构和路径，LLM 提供项目语义与假设，动态执行提供事实。

      第二，项目级上下文是核心瓶颈。 真实仓库无法依赖函数级分类或全文 Prompt。攻击面、可达性、程序切片、CPG 和高层程序分析工具接口正在成为重要基础设施。

      第三，验证正在取代发现成为主要难点。 现代模型可以快速生成大量候选，但真正证明攻击者可控、路径可达、缺少有效防御并产生实际安全影响仍然困难。

      第四，自动漏洞利用需要分级。 Crash、PoV、Exploit Primitive 和 Full Exploit 是不同能力层级。不同漏洞类型也需要不同的确定性 Oracle。

      第五，Harness 正在成为竞争核心。 2026 年的 OpenAnt、VVAH、Cloudflare Harness、Anthropic Defending Code Reference Harness、RAPTOR、Shannon、Taskflow Agent 等工作共同说明，工具编排、上下文路由、状态管理、验证协议和反馈闭环可能比单一模型选择更重要。

      第六，可复现性和证据链是当前系统的重要研究空白。 事件溯源、Artifact Hash、容器环境、动态日志、差分版本和 Replay 应当成为高质量自动化安全研究的重要评价维度。

      最后，从研究趋势看，未来一段时间最值得持续跟踪的方向包括：

      • Program Graph + LLM
      • LLM 自动生成程序分析规格
      • 攻击面与可达性驱动上下文
      • 独立对抗验证
      • 漏洞可利用性分级
      • Deterministic Runtime Oracle
      • LLM 自动生成并闭环优化 Fuzz Harness
      • 跨仓库依赖可达性
      • 发现—PoC—Patch—Regression 全生命周期
      • 事件溯源式可重放漏洞研究。

      这些方向共同反映了领域正在从“AI 给出一个漏洞答案”走向“AI 在工具和证据约束下完成可验证的漏洞研究过程”。
