# 使用说明

本文说明 agentic-code-audit 的本地运行、Docker 部署、任务创建、报告查看和开发测试方式。

## 1. 环境要求

推荐使用 Docker Compose 运行完整系统。开发调试时可单独启动后端和前端。

| 组件 | 用途 |
|---|---|
| Docker Desktop / Docker Compose | 启动 backend、frontend、sandbox 容器 |
| Git | 克隆被审计仓库 |
| Python 3.11 | 后端开发、本地 CLI 和测试 |
| Node.js 20 | 前端开发和生产构建 |
| DeepSeek API Key | LLM 策略、候选分析、验证方案和报告生成 |

## 2. 配置

复制环境变量模板：

```powershell
cd C:\Users\fujs\Desktop\security-agent\agentic-code-audit
Copy-Item .env.example .env
notepad .env
```

至少需要配置：

```env
DEEPSEEK_API_KEY=your-deepseek-api-key
DEEPSEEK_BASE_URL=https://api.deepseek.com
DEEPSEEK_MODEL=deepseek-v4-pro
```

`.env` 已加入 `.gitignore`，不要提交真实 Key。

## 3. Docker Compose 运行

启动：

```powershell
docker compose up -d --build
```

查看状态：

```powershell
docker compose ps
docker compose logs -f backend
```

停止：

```powershell
docker compose down
```

访问地址：

| 服务 | 地址 |
|---|---|
| 前端界面 | `http://127.0.0.1:3000` |
| 后端健康检查 | `http://127.0.0.1:8000/api/health` |

Compose 默认包含三类容器：

| 容器 | 作用 |
|---|---|
| `frontend` | React/Vite Web 界面 |
| `backend` | FastAPI、任务编排、数据库、报告服务 |
| `sandbox` | 无网络分析/验证环境，运行 cppcheck、clang-tidy、ctags、cmake、编译器和 harness |

## 4. 创建审计任务

前端新建调查时填写 GitHub URL 或本地路径，选择审计模式并点击开始。审计模式影响候选数量、LLM 调用预算和动态验证预算：

| 模式 | 特点 |
|---|---|
| `quick` | 快速扫描，候选和验证预算较小 |
| `standard` | 默认模式，平衡速度与覆盖 |
| `deep` | 更大候选预算和更多 LLM 复核，耗时更长 |

“允许 C/C++ native build”开关控制是否允许系统在 sandbox 中构建 C/C++ 项目。关闭时，即使工具可用，也不会执行原生构建。

## 5. CLI 运行

安装本地包：

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -e .
```

审计本地项目：

```powershell
agentic-code-audit audit .\examples\vulnerable-python -o reports\demo
```

审计 GitHub 仓库：

```powershell
agentic-code-audit audit https://github.com/Exiv2/exiv2.git -o reports\exiv2
```

带运行目标 URL 做 HTTP 动态验证：

```powershell
agentic-code-audit audit .\examples\vulnerable-python -o reports\demo --runtime-url http://127.0.0.1:5000
```

## 6. 审计输出

每次任务会在 `reports/<task-id>/` 下生成报告和证据：

| 文件/目录 | 内容 |
|---|---|
| `audit-report.json` | 完整结构化报告，前端主要读取该文件 |
| `audit-report.md` | Markdown 报告 |
| `mining-debug.json` | 挖掘阶段调试数据，包括 anchors、slices、candidates 和 findings 规模 |
| `pocs/` | PoC、runbook、verification.json、harness 和执行证据 |
| `artifacts/` | 工具输出、stdout/stderr、构建日志等证据文件 |

如果从其他机器复制报告，需要复制完整的 `reports/<task-id>/` 目录，而不是只复制 Markdown。前端历史记录依赖 `audit-report.json` 和任务目录结构。

## 7. 开发与测试

后端测试：

```powershell
python -m pytest
```

前端构建：

```powershell
cd frontend
npm install
npm run build
```

Compose 配置检查：

```powershell
docker compose config
```

## 8. MCP 与工具扩展

项目保留 MCP server 入口，用于外部工具或其他 Agent 编排系统接入。普通运行不依赖 MCP。

新增工具时优先通过 `ToolRunner` 统一接入，避免在 Agent 中直接调用本机命令。工具结果应保存 command、exit code、stdout、stderr、parsed artifact，并明确运行位置是 backend 还是 sandbox。
